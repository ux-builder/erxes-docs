# 개요

#erxes 커뮤니티에 오신 것을 환영합니다 👋

erxes는 투명성을 강조하는 커뮤니티 지향 프로젝트입니다. 우리는 우리의 비전을 공유하고 커뮤니티와 함께 erxes의 미래를 구축하기 위해 최선을 다하고 있습니다. erxes 로드맵은 커뮤니티 구성원들이 프로젝트에 대한 통찰을 얻고 올바른 방향으로 참여할 수 있도록 열려 있습니다.

여러분의 아이디어와 의견을 항상 환영합니다!

# 기여하는 방법
- 우리는 풀 리퀘스트를 환영하지만, 기여하기 전에 몇 가지 지침이 있습니다. 프로젝트에 풀 리퀘스트를 제출하기 전에 [기여 가이드](#)를 읽어주세요.
- 커뮤니티 성장과 투자를 위해 이 [투자](#) 기회를 공유하게 되어 기쁩니다.
- 개발한 플러그인을 배포하고 수익을 올릴 수 있는 [마켓플레이스](#)가 있습니다. 어떻게 이뤄낼 수 있는지 보여주는 [완전한 가이드라인](#)이 있습니다. 이 가이드는 좋아하는 일을 하며 돈을 벌고 싶은 사람들을 위한 것입니다.

# 코드베이스에 기여하기

erxes에 기여해주셔서 감사합니다! 이 문서는 이 저장소에 변경 사항을 제출하는 방법과 따를 규칙을 설명합니다. 궁금한 점이 있으면 여기에 이슈를 제출하거나 [Discord](https://discord.com/invite/aaGzy3gQK5)를 통해 문의해 주세요.

## 사전 요구사항

- GitHub 이슈 및 풀 리퀘스트에 익숙해야 합니다.
- 문서를 읽어야 합니다.
- erxes로 테스트 프로젝트를 설정해야 합니다.
## PR 이전에 이슈 생성

1. 변경 작업을 시작하기 전에 작업할 내용에 대한 이슈가 있는지 확인해 주세요. [기존 이슈](https://github.com/erxes/erxes/issues)를 찾거나 [새 이슈를 열어](https://github.com/erxes/erxes/issues/new/choose) 이슈가 없는 경우 열어 주세요. 이렇게 하면 다른 사람들이 의견을 제시하거나 대안을 제안할 수 있어 erxes의 미래에 가장 적합한 변경만 추가되도록 할 수 있습니다.
2. 변경 작업을 시작할 준비가 되면 먼저 [erxes 저장소를 포크](https://docs.github.com/en/get-started/quickstart/fork-a-repo)하고 [브랜치를 생성](https://docs.github.com/en/pull-requests/collaborating-with-pull-requests/proposing-changes-to-your-work-with-pull-requests/creating-and-deleting-branches-within-your-repository)하세요. develop 브랜치에서 시작하세요.
3. 변경 작업을 수행하세요.
4. [erxes 저장소의 develop 브랜치를 대상으로 풀 리퀘스트를 생성](https://docs.github.com/en/pull-requests/collaborating-with-pull-requests/proposing-changes-to-your-work-with-pull-requests/creating-a-pull-request-from-a-fork)하세요. 며칠 내에 erxes 팀원이 PR을 검토하고 의견을 제시하며 최종적으로 승인할 것입니다.
# 작업 흐름
### 브랜치

모든 변경 사항은 브랜치의 일부여야 하며 풀 리퀘스트로 제출해야 합니다. 브랜치는 다음 중 하나로 접두어를 붙여야 합니다:

- fix/ - 버그 수정
- feat/ - 기능 추가
- docs/ - 문서 변경

### 커밋

커밋은 작고 독립적으로 유지하려고 노력하세요. 이렇게 하면 리뷰어가 내용을 이해하기 쉽고 요청을 처리하기 쉬워집니다.

### 풀 리퀘스트

변경 작업이 완료되면 브랜치를 풀 리퀘스트로 제출해야 합니다. 풀 리퀘스트는 erxes 저장소의 develop 브랜치를 대상으로 열어야 합니다.

PR 설명에는 다음 구조를 따라야 합니다:

- 무엇 - 이 PR에 포함된 변경 사항
- 왜 - 이러한 변경이 왜 중요한지
- 어떻게 - 변경 사항이 어떻게 구현되었는지
- 테스트 - 변경 사항이 어떻게 테스트되었는지 또는 리뷰어가 기능을 테스트하는 방법

리뷰 요청 전에 자체 검토를 강력히 권장합니다. 자체 검토를 하려면 오른쪽 상단의 검토 버튼을 클릭하고 코드를 검토하여 변경 사항에 주석을 달아주세요. 이렇게 하면 리뷰어가 PR을 처리하기 쉬워집니다.

### 병합 스타일

모든 풀 리퀘스트는 스쿼시(Squash) 병합됩니다.

### 테스트

모든 PR에는 포함된 변경 사항에 대한 테스트가 포함되어야 합니다. 우리는 다음 두 가지 유형의 테스트를 작성해야 합니다:

- packages/*/src/services/**tests** 및 packages/*/src/api/routes/*/**tests**에 있는 단위 테스트
- integration-tests/*/**tests**에 있는 통합 테스트

# 문서화

- 일반적으로 코드에 주석을 추가하여 변경 사항을 문서화하는 것을 권장합니다.
- 사용자에게 영향을 미치는 동작을 변경하는 경우 해당 변경 사항에 대한 문서를 제공해야 합니다.
- 모든 메서드 및 엔드포인트는 [JSDoc](https://jsdoc.app/) 및 [swagger-inline](https://www.npmjs.com/package/swagger-inline)을 사용하여 문서화해야 합니다.

<Note>
  나중에 erxes 코드베이스에 대한 변경 사항을 문서화하는 데 기여하는 경우, 
  문서 웹사이트의 기여 지침도 확인하세요.
</Note>

# 사이트 설정

문서 웹사이트는 문서 생성 최적화 프레임워크인 [Docusaurus](https://docusaurus.io/)로 구축되었습니다. Docusaurus에 익숙하지 않다면, Docusaurus가 어떻게 작동하고 구조와 세부 정보를 이해하기 위해 [설치 문서](https://docusaurus.io/docs/installation)를 확인하는 것이 좋습니다. 문서 코드베이스는 GitHub의 [erxes 저장소](https://github.com/erxes/erxes)에 호스팅되어 있습니다. Docusaurus 웹사이트를 실행하는 코드는 www/docs 디렉토리에서 찾을 수 있습니다.

# 문서 내용

문서 내용은 Markdown 형식으로 작성됩니다. Markdown에 익숙하지 않다면, 빠른 시작을 위한 [치트 시트](https://www.markdownguide.org/cheat-sheet/)를 확인하세요. 또한 MDX 파일도 있습니다. MDX 파일은 Markdown의 강력함과 React를 결합한 것입니다. 따라서 파일의 내용에는 JSX 컴포넌트와 import 문이 포함될 수 있습니다. [Docusaurus의 가이드](https://docusaurus.io/docs/markdown-features/react)에서 MDX에 대해 더 알아볼 수 있습니다.

# 기여 방법

기존 문서 페이지의 오류를 수정하는 경우, 페이지 하단으로 스크롤하여 "이 페이지 편집" 링크를 클릭하세요. 그러면 해당 페이지의 GitHub 편집 폼으로 리디렉션되어 직접 수정하고 풀 리퀘스트(PR)를 제출할 수 있습니다. 새 페이지를 추가하거나 코드베이스에 기여하는 경우, 저장소를 포크하고 새 브랜치를 생성하여 필요한 모든 변경 작업을 수행합니다. 그런 다음, erxes 저장소에 PR을 생성하세요. 기여 방법에 대한 자세한 내용은 저장소의 기여 지침을 확인하세요.

- 브랜치 이름 기존 문서 페이지를 수정하거나 문서 변경을 위해 저장소를 포크할 때는 새 브랜치를 생성해야 합니다. 브랜치 이름은 docs/로 시작해야 합니다. 예를 들어, docs/fix-services와 같이 작성합니다.

- 풀 리퀘스트 규칙 풀 리퀘스트를 생성할 때 제목에 "docs:"를 접두사로 붙이세요. "docs"는 소문자로 유지하세요. PR 본문에는 PR이 하는 일을 명확하게 설명하세요. PR이 이슈를 해결하는 경우, 이슈 번호와 함께 닫기 키워드를 사용하세요. 예를 들어, "Closes #1333"와 같이 작성합니다.

# 테이블

콜론을 사용하여 열을 정렬할 수 있습니다.

| 테이블 | 는 | 멋집니다 |
| ---- | ---- | ----------- |
| col 3은 | 오른쪽 정렬됨 | \$1600 |
| col 2는 | 가운데 정렬됨 | $12 |
| 얼룩무늬 | 멋짐 | $1 |

각 헤더 셀을 구분하는 최소한 3개의 대시가 있어야 합니다. 외부 파이프(|)는 선택 사항이며, 원시 Markdown 줄을 예쁘게 정렬할 필요는 없습니다. 인라인 Markdown도 사용할 수 있습니다.

| Markdown | 덜 | 예쁨 |
| ---- | ---- | ----------- |
| *여전히* | `렌더링` | 잘됨 |
| 1 | 2 | 3 |
